Apache Maven

Apache Maven – фреймворк для автоматизации сборки проектов. Кроме сборки проектов Maven позволяет сгенерировать каркас проекта и управлять зависимостями внутри проекта и импортом сторонних библиотек из своего огромного репозитория.

Зачем он вообще?

Ну вот например мы написали простую программу в файле HelloWorld.java. Далее нам ее нужно скомпилировать с помощью JAVAC (java-компилятора), после чего мы получим файл HelloWorld.class. И дажее уже этот файл мы можем запускать на любой JVM, написанной для любой операционки. И казалось бы всё ок, но а если у нас тысячи таких файлов. Хорошо бы автоматизировать этот процесс. Поэтому одной из главных задач MAVEN является отправка кода на компиляцию и последующее распределение всех файлов по папкам и последующей упаковки этох файлов и папок в архив (war или jar).

Каждый проект, для того чтобы не изобретать велосипед заново, использует различные библиотеки и фреймоврки, то есть зависит от них. И MAVEN помогает управлять этими зависимостями, мы указываем ему какие нам нужны, и он сам их подкачивает из удаленного репозитория.

Maven еще имеет большое количество полезных функций которые упрощают жизнь разработчка, которые описаны в его жизненном цикле.

Жизненный цикл (фазы сборки проекта)

  • сlean — Maven удаляет директорию сборки проекта со всеми скомпилированными файлами.
  • validate – проверяет корректность метаинформации о проекте, является ли структура проекта полной и правильной.
  • compile – компилируются исходного кода.
  • test – фаза тестирования собранного кода, заранее подготовленным набором тестов.
  • package – упаковка откомпилированных классов и прочих ресурсов. Например, в JAR-файл.
  • verify – эта фаза сама по себе ничего неделает, на нее могут назначаться плагины, попадающие под описание: "выполнение любых проверок для верификации пакета на валидность и соответствие критериям качества".
  • install – установка программного обеспечения в локальный Maven-репозиторий, чтобы сделать его доступным для других проектов текущего пользователя.
  • site – жизненный цикл генерации проектной документации.
  • deploy – стабильная версия программного обеспечения распространяется на удаленный Maven-репозиторий, чтобы сделать его доступным для других пользователей.

pom.xml

Информация для сборки проекта находится в pom.xml файле. При запуске Maven проверяет, содержит ли конфигурационный файл все необходимые данные и все ли данные синтаксически правильно записаны.
Минимальная конфигурация включает версию конфигурационного файла, имя проекта, его автора и версию. С помощью pom.xml конфигурируются зависимости от других проектов, а также список плагинов, реализующий порядок сборки.
Также pom.xml может быть унаследован от другой pom теге parent.
Ключевым понятием Maven является артефакт — это, по сути, любая библиотека, хранящаяся в репозитории. Это может быть какая-то зависимость или плагин.
Зависимости — это те библиотеки, которые непосредственно используются в вашем проекте для компиляции кода или его тестирования.
Плагины же используются самим Maven'ом при сборке проекта или для каких-то других целей (деплоймент, создание файлов проекта для Eclipse и др.).
.m2 - папка куда буду выкачиваться наши зависимости
settings.xml - конфигурационный файл указывающий мавену из каких испточников качать br>

Плагины

При сборке проектов, часто возникают задачи для более точной настройки того или иного этапа жизненного цикла сборки. Например: конфигурирование компилятора, создание jar-файла и файла манифеста, копирования ресурсов и тд.

  • maven-compiler-plugin – Это, пожалуй, самый популярный плагин, позволяющий управлять версией компилятора. Настройки по умолчанию в мавен далеки от идеала и редкий проект обходится без конфигурирования параметров компилятора. В простейшем случае плагин позволяет задать версию java для которой написан код и версию java для компилирования. Плагин позволяет указать путь к компилятору, запустить компилирование в отдельной JVM, задавать аргументы компилирования и даже выполнить компилирование не-java компилятором.
  • maven-jar-plugin – Этот плагин позволяет убить сразу нескольких зайцев: создать манифест и jar-файл, и описать дополнительные ресурсы, которые необходимо включить в jar-файл. Дополнительные ресурсы, включаемые в jar, описываются тегами include и exclude (для включения и исключения соответственно).
  • maven-dependency-plugin – Плагин позволяет копировать зависимости, описанные в pom.xml в указанную директорию.
  • maven-jar-plugin – Помимо копирования ресурсов во время сборки проекта, я пользовался этим плагином на стадии инсталляции, для копирования проекта на сервер.

    • Структура папок

    При работе с maven необходимо соблюдать соглашение о стандартной стуктуре папок:

    • Корневой каталог проекта: файл pom.xml и все дальнейшие подкаталоги
    • src: все исходные файлы
    • src/main: исходные файлы собственно для продукта
    • src/main/java: Java-исходный код
    • src/main/resources: другие файлы, которые используются при компиляции или исполнении, например Properties-файлы
    • src/main/webapp/WEB-INF/web.xml – файл конфигурации web-проекта
    • src/test: исходные файлы, необходимые для организации автоматического тестирования
    • src/test/java: JUnit-тест-задания для автоматического тестирования
    • target: все создаваемые в процессе работы Мавена файлы
    • target/classes: компилированные Java-классы
