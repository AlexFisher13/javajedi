Системы контроля версий

Начало контроля версий

Многие предпочитают контролировать версии, просто копируя файлы в другой каталог (как правило добавляя текущую дату к названию каталога). Такой подход очень распространён, потому что прост, но он и чаще даёт сбои.

Локальные системы контроля версий

Чтобы решить эту проблему сбоев, программисты уже давно разработали локальные СКВ с простой базой данных, в которой хранятся все изменения нужных файлов. Одна из поулярных таких СКВ является rcs. Ее работа основана на системе патчей, в файл записываются данные об изменениях. Это повзоляет пересоздать любой файл на любой момент времени, последовательно накладывая патчи.

Централизованные системы контроля версий

Для того чтобы многие разработчики могли участвовать в проекте придумали централизованные СКВ. У администраторов появился чёткий контроль над тем, кто и что может делать, и, конечно, администрировать ЦСКВ намного легче, чем локальные базы на каждом клиенте. Однако при таком подходе есть и несколько серьёзных недостатков. Наиболее очевидный — централизованный сервер является уязвимым местом всей системы. Если сервер выключается на час, то в течение часа разработчики не могут взаимодействовать, и никто не может сохранить новой версии своей работы. А если сервер повреждается то теряется абсолютно все.

Распределенные системы контроля версий

Лучшие представители этого вида это Git, Mercurial, Bazaar или Darcs клиенты не просто выгружают последние версии файлов, а полностью копируют весь репозиторий. Поэтому в случае, когда "умирает" сервер, через который шла работа, любой клиентский репозиторий может быть скопирован обратно на сервер, чтобы восстановить базу данных. Каждый раз, когда клиент забирает свежую версию файлов, он создаёт себе полную копию всех данных.

Система контроля версий дает возможность:

  • возвращать к прежнему состоянию весь проект, или отдельные файлы
  • просматривать происходящие со временем изменения
  • определять, кто и какие изменения вносил в проект

Пользуясь СКВ, даже если вы всё испортите или потеряете файлы, всё можно будет легко восстановить.

Git

Инициализация

  • Инициализация гита в проекте: git init
  • Установка имени пользователя: git config --global user.name "Ivan"
  • Посмотреть статус проекта: git status
  • Создание .gitignore файла:
    *.iml
    .idea/*
    *.class
    *.war
    *.jar
    *.ear
    target/

Внесение изменений в проект

  • Добавление файла в гит: git add pom.xml
  • Добавление всех файлов: git add --all
  • Коммит изменений: git commit -m "fix bugs"

Отправка изменений на удаленный репозиторий

  • Создаем новый репозиторий, наример на GitHub
  • Копируем адрес созданного репозитория, например https://github.com/ivan/helloworld (без .git)
  • Привязываем наш проект к удаленному репозиторию: git remote add helloproject https://github.com/ivan/helloworld
  • Для просмотра всех удаленных репозиториев: git remote
  • Пушим изменения в репозиторий в ветку мастер: git push fisherproject master

Работа с ветками

  • Создаем новую версию проекта: git tag "version_1.0"
  • Создаем ветку от данной версии: git branch experimentalBranch
  • Посмтреть какие ветки есть: git branch
  • Переместиться на ветку experimentalBranch: git checkout experimentalBranch
  • Смержить текующую ветку с experimentalBranch: git merge experimentalBranch
